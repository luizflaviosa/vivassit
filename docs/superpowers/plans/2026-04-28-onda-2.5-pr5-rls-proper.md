# Onda 2.5 PR5 — RLS Proper para tenants/tenant_doctors/tenant_api_keys

**Goal:** Completar o PR5 da Onda 2.5 adicionando policies de INSERT/UPDATE/DELETE (e SELECT onde faltava) nas tabelas multi-tenant críticas, garantindo defesa em profundidade contra vazamento cross-tenant mesmo se app code esquecer filtro de tenant_id.

**Architecture:** Service_role (`supabaseAdmin()`) continua bypassando RLS — todas as rotas atuais funcionam sem mudança. Policies novas protegem leituras/escritas via session token (anon key + JWT do usuário), bloqueando cross-tenant leak. Helpers SQL `is_tenant_member(t)` (já existe) e `is_tenant_admin(t)` / `is_tenant_owner(t)` (novos) padronizam checks.

**Risk:** Baixo. App layer não depende de policies hoje (só service_role). Migração apenas adiciona — não remove nem altera comportamento existente. Se uma policy quebrar algo, drop individual a remove em <1s.

**Helper existente:**
```sql
CREATE FUNCTION is_tenant_member(t varchar) RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = 'public', 'pg_temp'
AS $$ SELECT EXISTS (
  SELECT 1 FROM public.tenant_members
  WHERE user_id = auth.uid() AND tenant_id = t AND status = 'active'
) $$;
```

**Dados:** 5 tenants, 2 com admin_user_id, 2 tenant_members (ambos role='owner', status='active'), 2 tenant_doctors, 0 tenant_api_keys.

---

## Decisões de design

1. **Service_role bypassa tudo** — automático no Supabase. Não precisa policy explícita pro role, mas vou usar `TO authenticated` em policies pra deixar intent claro (anon nunca consegue).

2. **`tenant_api_keys` permanece restrito** — mesmo dono não consegue ler valor decifrado via session. App layer expõe só metadata (service_name, status, last_rotated) usando service_role. Policy nova permite SELECT/manage só se necessário; por enquanto continua só service_role (zero policies = deny por padrão), mas a migração documenta isso explicitamente com comentário SQL.

3. **`tenants` UPDATE = só owner** — admin não muda clinic_name, plan_type, etc. Owner sim. Campos sensíveis (subscription_status, plan_type) em teoria deveriam ter restrição mais fina, mas isso é refinamento futuro — hoje o app layer já controla via select de colunas no GET/PATCH.

4. **`tenant_doctors`:** owner/admin podem CRUD; doctor pode UPDATE própria linha (`linked_user_id = auth.uid()`).

5. **`tenant_members`:** owner pode INSERT/UPDATE/DELETE; member pode UPDATE própria linha (telegram_chat_id, etc.).

6. **WITH CHECK = USING** em UPDATEs — impede que owner mude `tenant_id` da linha pra outro tenant.

## Migration SQL

```sql
-- ============================================================
-- Onda 2.5 PR5: RLS proper em tenants/tenant_doctors/tenant_api_keys/tenant_members
-- Adds INSERT/UPDATE/DELETE policies (read policies já existiam).
-- Service_role bypassa tudo automaticamente — esta migration NÃO afeta as rotas
-- /api/* atuais que usam supabaseAdmin().
-- ============================================================

-- 1) Helper: is_tenant_admin (role IN owner|admin)
CREATE OR REPLACE FUNCTION public.is_tenant_admin(t varchar)
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = 'public', 'pg_temp'
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.tenant_members
    WHERE user_id = auth.uid()
      AND tenant_id = t
      AND role IN ('owner', 'admin')
      AND status = 'active'
  );
$$;

-- 2) Helper: is_tenant_owner (role = owner)
CREATE OR REPLACE FUNCTION public.is_tenant_owner(t varchar)
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = 'public', 'pg_temp'
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.tenant_members
    WHERE user_id = auth.uid()
      AND tenant_id = t
      AND role = 'owner'
      AND status = 'active'
  );
$$;

GRANT EXECUTE ON FUNCTION public.is_tenant_admin(varchar) TO authenticated;
GRANT EXECUTE ON FUNCTION public.is_tenant_owner(varchar) TO authenticated;

-- ============================================================
-- TENANTS
-- ============================================================
-- SELECT já existe (tenants_member_read). Adicionando UPDATE.
-- INSERT e DELETE: deny pra authenticated (só service_role cria/deleta).

CREATE POLICY tenants_owner_update ON public.tenants
  FOR UPDATE TO authenticated
  USING (public.is_tenant_owner(tenant_id))
  WITH CHECK (public.is_tenant_owner(tenant_id));

-- ============================================================
-- TENANT_DOCTORS
-- ============================================================
-- SELECT já existe. Adiciona INSERT/UPDATE/DELETE.

CREATE POLICY tenant_doctors_admin_insert ON public.tenant_doctors
  FOR INSERT TO authenticated
  WITH CHECK (public.is_tenant_admin(tenant_id));

CREATE POLICY tenant_doctors_admin_update ON public.tenant_doctors
  FOR UPDATE TO authenticated
  USING (public.is_tenant_admin(tenant_id))
  WITH CHECK (public.is_tenant_admin(tenant_id));

-- Doctor pode atualizar a própria linha (ex: assistant_prompt, contact_email)
CREATE POLICY tenant_doctors_self_update ON public.tenant_doctors
  FOR UPDATE TO authenticated
  USING (linked_user_id = auth.uid())
  WITH CHECK (linked_user_id = auth.uid());

CREATE POLICY tenant_doctors_owner_delete ON public.tenant_doctors
  FOR DELETE TO authenticated
  USING (public.is_tenant_owner(tenant_id));

-- ============================================================
-- TENANT_API_KEYS
-- ============================================================
-- Política deliberada: ZERO policies = bloqueado pra anon/authenticated.
-- Apenas service_role acessa (rotas /api/marketplace/* via supabaseAdmin()).
-- API keys são credenciais sensíveis — não devem ser legíveis via session token,
-- mesmo pelo owner. Se no futuro precisarmos UI pra owner ver metadata
-- (service_name, last_rotated), criar uma view específica que omite
-- api_key_encrypted, NÃO uma policy de SELECT na tabela base.

COMMENT ON TABLE public.tenant_api_keys IS
  'Credenciais sensíveis. RLS habilitado SEM policies — apenas service_role acessa. Não adicionar SELECT policy direta; criar view se UI precisar de metadata.';

-- ============================================================
-- TENANT_MEMBERS
-- ============================================================
-- SELECT já existe (members_self_read + members_same_tenant_read).
-- Adiciona INSERT/UPDATE/DELETE.

-- Owner convida novos members
CREATE POLICY tenant_members_owner_insert ON public.tenant_members
  FOR INSERT TO authenticated
  WITH CHECK (public.is_tenant_owner(tenant_id));

-- Owner gerencia members do tenant; member atualiza própria linha
CREATE POLICY tenant_members_owner_update ON public.tenant_members
  FOR UPDATE TO authenticated
  USING (public.is_tenant_owner(tenant_id))
  WITH CHECK (public.is_tenant_owner(tenant_id));

CREATE POLICY tenant_members_self_update ON public.tenant_members
  FOR UPDATE TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

-- Owner remove members; member sai sozinho (soft delete preferível, mas DELETE permitido)
CREATE POLICY tenant_members_owner_delete ON public.tenant_members
  FOR DELETE TO authenticated
  USING (public.is_tenant_owner(tenant_id));

CREATE POLICY tenant_members_self_delete ON public.tenant_members
  FOR DELETE TO authenticated
  USING (user_id = auth.uid());
```

## Plano de teste pré-aplicação

Rodar em transação com `BEGIN; ... ROLLBACK;` simulando 3 personas:

1. **Service role** (como rotas /api/* hoje) → tudo deve funcionar
2. **Owner do tenant A** (auth.uid() = um user_id em tenant_members) → vê só seu tenant; UPDATEs no próprio tenant funcionam; tenta ler tenant B → 0 linhas
3. **Anon** (auth.uid() IS NULL) → tudo bloqueado

## Rollback

Se qualquer coisa quebrar:

```sql
-- Drop policies novas (mantém as antigas funcionando)
DROP POLICY IF EXISTS tenants_owner_update ON public.tenants;
DROP POLICY IF EXISTS tenant_doctors_admin_insert ON public.tenant_doctors;
DROP POLICY IF EXISTS tenant_doctors_admin_update ON public.tenant_doctors;
DROP POLICY IF EXISTS tenant_doctors_self_update ON public.tenant_doctors;
DROP POLICY IF EXISTS tenant_doctors_owner_delete ON public.tenant_doctors;
DROP POLICY IF EXISTS tenant_members_owner_insert ON public.tenant_members;
DROP POLICY IF EXISTS tenant_members_owner_update ON public.tenant_members;
DROP POLICY IF EXISTS tenant_members_self_update ON public.tenant_members;
DROP POLICY IF EXISTS tenant_members_owner_delete ON public.tenant_members;
DROP POLICY IF EXISTS tenant_members_self_delete ON public.tenant_members;

```
